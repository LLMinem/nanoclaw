This task executes after the user provides the Telegram bot token in Task 1. Per D-11, explain each step.

**Step 1 — Add bot token to .env and sync to container:**
Add `TELEGRAM_BOT_TOKEN=<user-provided-token>` to `.env` file.
Then sync:

```bash
cp .env data/env/env
```

**Step 2 — Verify bot token with Telegram API:**

```bash
curl -s "https://api.telegram.org/bot${TELEGRAM_BOT_TOKEN}/getMe" | python3 -m json.tool
```

Expected: JSON with `"ok": true` and the bot's username and id.

**Step 3 — Build the project (ensure latest code is compiled):**

```bash
npm run build
```

**Step 4 — Create NanoClaw systemd user service (per D-12):**

```bash
mkdir -p ~/.config/systemd/user
```

Create `~/.config/systemd/user/nanoclaw.service`:

```ini
[Unit]
Description=NanoClaw AI Assistant (Fritz)
After=network.target onecli.service docker.service
Wants=onecli.service

[Service]
Type=simple
WorkingDirectory=/path/to/nanoclaw
ExecStart=/usr/bin/node dist/index.js
Restart=on-failure
RestartSec=10
StandardOutput=journal
StandardError=journal
Environment=NODE_ENV=production
Environment=LOG_LEVEL=info

[Install]
WantedBy=default.target
```

**IMPORTANT:** Replace `/path/to/nanoclaw` with the actual project root path (output of `pwd`). Replace `/usr/bin/node` with the actual Node.js binary path (output of `which node`).

Enable and start:

```bash
systemctl --user daemon-reload
systemctl --user enable nanoclaw
systemctl --user start nanoclaw
```

Enable lingering so user services survive logout:

```bash
loginctl enable-linger $(whoami)
```

Verify service started:

```bash
systemctl --user status nanoclaw
journalctl --user -u nanoclaw --no-pager -n 20
```

Look for startup logs showing channel connections. The Telegram channel should connect and log something like "Telegram bot started" or similar.

**Step 5 — Get the chat ID for registration:**
The user needs to send `/chatid` to the Fritz bot in Telegram. For a 1:1 DM (per D-01), this should return their chat ID in the format `tg:<number>`.

Wait for the service to be running and the bot to be responding to `/chatid`. Check logs:

```bash
journalctl --user -u nanoclaw -f
```

Once the user provides the chat ID, proceed to registration.

**Step 6 — Register the 1:1 DM as main group (per D-01, D-03):**

```bash
npx tsx setup/index.ts --step register -- \
  --jid "tg:<CHAT_ID>" \
  --name "Fritz DM" \
  --folder "telegram_main" \
  --trigger "@Fritz" \
  --channel telegram \
  --no-trigger-required \
  --is-main
```

Replace `<CHAT_ID>` with the user's Telegram chat ID from step 5.

- `--no-trigger-required` — main group responds to all messages without @mention prefix
- `--is-main` — marks this as the control group with elevated privileges

**Step 7 — Configure sender allowlist (per D-04, TELE-05):**
Create `~/.config/nanoclaw/sender-allowlist.json`:

```bash
mkdir -p ~/.config/nanoclaw
```

Write the allowlist file with only meeq's Telegram user ID:

```json
{
  "default": {
    "allow": ["<MEEQ_TELEGRAM_USER_ID>"],
    "mode": "drop"
  },
  "chats": {},
  "logDenied": true
}
```

The Telegram user ID is available from the `/chatid` response (for 1:1 DMs, chat ID = user ID). The format is just the numeric ID as a string, e.g., `"123456789"`.

**`"mode": "drop"`** means messages from non-allowed senders are silently dropped (not just trigger-filtered). This is the strictest mode — only meeq can interact with the bot at all.

**Step 8 — Create mount allowlist (empty — no external directories needed yet):**

```bash
cat > ~/.config/nanoclaw/mount-allowlist.json << 'EOF'
{
  "allowedRoots": [],
  "blockedPatterns": [],
  "nonMainReadOnly": true
}
EOF
```

**Step 9 — Restart service to pick up registration and allowlist:**

```bash
systemctl --user restart nanoclaw
```

Wait 5 seconds, then check status:

```bash
systemctl --user status nanoclaw
journalctl --user -u nanoclaw --no-pager -n 20
```

<threat_model>

## Trust Boundaries

| Boundary                         | Description                                 |
| -------------------------------- | ------------------------------------------- |
| Telegram users → NanoClaw bot    | Untrusted messages from any Telegram user   |
| NanoClaw host → Docker container | Prompt data crosses into isolated container |
| systemd → NanoClaw process       | Service management boundary                 |

## STRIDE Threat Register

| Threat ID | Category              | Component                  | Disposition | Mitigation Plan                                                                                                                                                                      |
| --------- | --------------------- | -------------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| T-01-07   | S (Spoofing)          | Telegram sender            | mitigate    | Sender allowlist at `~/.config/nanoclaw/sender-allowlist.json` restricts to meeq's Telegram ID only. `mode: "drop"` silently discards unauthorized messages.                         |
| T-01-08   | I (Info Disclosure)   | TELEGRAM_BOT_TOKEN in .env | mitigate    | `.env` is read by selective reader (`src/env.ts`) — token is used only by host-side Telegram channel, never passed to containers. File permissions should be 600 (`chmod 600 .env`). |
| T-01-09   | T (Tampering)         | systemd service file       | accept      | User-level systemd service — only the owning user can modify. Standard Linux permission model.                                                                                       |
| T-01-10   | D (Denial of Service) | Telegram message flood     | mitigate    | GroupQueue limits concurrent containers (MAX_CONCURRENT_CONTAINERS=5). Exponential backoff on failures (5s base, max 5 retries).                                                     |
| T-01-11   | R (Repudiation)       | Message history            | mitigate    | All messages stored in SQLite database (`store/messages.db`) with timestamps. Service logs via journalctl provide audit trail.                                                       |

</threat_model>
