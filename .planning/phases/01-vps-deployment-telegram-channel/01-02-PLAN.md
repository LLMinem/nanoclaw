- `node:22-slim` base

Per D-05, D-06, and D-07: install OneCLI on the VPS and configure it as a systemd service.

**Step 1 — Install OneCLI gateway and CLI tool:**

```bash
curl -fsSL onecli.sh/install | sh
curl -fsSL onecli.sh/cli/install | sh
```

If `onecli version` fails after install, add `~/.local/bin` to PATH:

```bash
export PATH="$HOME/.local/bin:$PATH"
grep -q '.local/bin' ~/.bashrc 2>/dev/null || echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc
```

**Step 2 — Point CLI at local gateway:**

```bash
onecli config set api-host http://127.0.0.1:10254
```

**Step 3 — Verify gateway is running:**

```bash
curl -sf http://127.0.0.1:10254/health && echo "Gateway OK" || echo "Gateway not ready"
```

If not ready, wait up to 15 seconds:

```bash
for i in $(seq 1 15); do curl -sf http://127.0.0.1:10254/health && break; sleep 1; done
```

If still not ready: `onecli start` or check `ps aux | grep onecli`.

**Step 4 — Configure OneCLI as systemd service (per D-07):**
Check if OneCLI installed its own systemd unit. If not, the gateway process needs to auto-start on boot. Check:

```bash
systemctl --user list-unit-files | grep onecli || systemctl list-unit-files | grep onecli
```

If no systemd unit exists, OneCLI may manage its own persistence. Verify by rebooting test:

```bash
onecli version && curl -sf http://127.0.0.1:10254/health
```

If OneCLI doesn't have built-in systemd support, create a minimal user service:

```bash
mkdir -p ~/.config/systemd/user
cat > ~/.config/systemd/user/onecli.service << 'EOF'
[Unit]
Description=OneCLI Agent Vault Gateway
After=network.target

[Service]
Type=simple
ExecStart=%h/.local/bin/onecli serve
Restart=on-failure
RestartSec=5
Environment=HOME=%h

[Install]
WantedBy=default.target
EOF
systemctl --user daemon-reload
systemctl --user enable onecli
systemctl --user start onecli
```

Verify: `systemctl --user status onecli` should show active.

**Step 5 — Create/update .env file:**
Create or update `.env` in the project root with these values:

```
ASSISTANT_NAME=Fritz
ONECLI_URL=http://127.0.0.1:10254
TZ=Europe/Berlin
```

- `ASSISTANT_NAME=Fritz` — the bot's name (used for trigger pattern `@Fritz`)
- `ONECLI_URL` — points to local OneCLI gateway
- `TZ` — timezone for scheduled tasks and message formatting (Germany)

Do NOT add any API keys or tokens to `.env` — those go through OneCLI.

**Step 6 — Sync .env to container environment:**

```bash

cp .env data/env/env
```

The container reads its environment from `data/env/env`, not `.env` directly (per add-telegram SKILL.md).

<threat_model>

## Trust Boundaries

| Boundary                       | Description                                               |
| ------------------------------ | --------------------------------------------------------- |
| VPS host → Docker container    | Container isolation boundary — containers run as non-root |
| OneCLI gateway → Anthropic API | OAuth token injected into HTTPS traffic at gateway level  |
| User terminal → OneCLI         | OAuth token passes through CLI during registration        |

## STRIDE Threat Register

| Threat ID | Category                   | Component                 | Disposition | Mitigation Plan                                                                                                                   |
| --------- | -------------------------- | ------------------------- | ----------- | --------------------------------------------------------------------------------------------------------------------------------- |
| T-01-03   | I (Info Disclosure)        | OneCLI gateway            | mitigate    | OneCLI binds to 127.0.0.1:10254 (localhost only) — not exposed to network. Containers reach it via Docker host gateway.           |
| T-01-04   | S (Spoofing)               | Docker container → OneCLI | mitigate    | OneCLI injects credentials based on host-pattern matching (api.anthropic.com), preventing credential leakage to arbitrary hosts.  |
| T-01-05   | D (Denial of Service)      | Docker daemon             | accept      | Single-user VPS — Docker resource exhaustion is self-inflicted. MAX_CONCURRENT_CONTAINERS (default 5) limits parallel containers. |
| T-01-06   | E (Elevation of Privilege) | Container escape          | mitigate    | Container runs as non-root `node` user (uid 1000). No privileged capabilities. No Docker socket mounted.                          |

</threat_model>
